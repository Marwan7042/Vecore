CUDA Acceleration

How Vecore leverages your NVIDIA GPU for blazing-fast deep learning.

Overview

Vecore is designed from the ground up to run on NVIDIA GPUs. Every operation — from matrix multiplication to gradient descent — has a hand-written CUDA kernel optimized for throughput.

Device Management

  • .to("cuda") transfers tensors and models to GPU
  • .to("cpu") brings them back
Tensor a = Tensor::randn({1024, 1024});
a.to("cuda"); // Transfer to GPU

// Operations happen on the current device
Tensor b = Tensor::randn({1024, 1024}).to("cuda");
Tensor c = a.matmul(b);

c.to("cpu"); // Transfer back to host memory

cuBLAS Integration

  • Matrix multiplication uses NVIDIA cuBLAS
  • TF32 Tensor Core math enabled via cublasSetMathMode(handle, CUBLAS_TF32_TENSOR_OP_MATH)
  • Static thread-local cuBLAS handle eliminates initialization overhead
Tip: TF32 provides near-FP32 accuracy with 8x the throughput on Ampere+ GPUs.

Custom CUDA Kernels

Vecore implements custom optimized kernels for various operations. Below are some key kernels and their optimization techniques:

cuda_relu_kernel

Element-wise ReLU activation.

  • 256 threads per block
  • Uses cudaMemsetAsync for zero initialization

cuda_relu_backward_kernel

ReLU gradient computation.

  • Zero-copy: only passes gradient where input > 0

cuda_broadcast_add_kernel

Broadcasted element-wise addition.

  • Handles shape broadcasting across arbitrary dimensions
  • Flattened index computation with stride arrays

cuda_sgd_kernel

In-place SGD parameter update.

  • W -= lr * grad, runs asynchronously

cuda_cross_entropy_metrics_kernel

Fused loss + accuracy + gradient.

  • Shared memory block reduction (eliminates atomicAdd contention)
  • Computes loss, accuracy, and backprop gradient in a single kernel
  • Reduced atomic collisions by 99.6%

cuda_sum_kernel_atomic

Parallel reduction for sum operations.

Memory Management

CachingAllocator — GPU memory pool

  • Caches freed CUDA allocations by size
  • Reuses memory blocks to avoid cudaMalloc latency
  • allocate(size) checks pool before calling cudaMalloc
  • free(ptr, size) returns to pool instead of cudaFree

GPUData<T> — RAII wrapper

  • Constructor allocates via CachingAllocator
  • Destructor returns to pool
  • Prevents memory leaks

Async Pipeline (Advanced)

Sequential::fit utilizes a double-buffered pipeline to hide data transfer latency:

  • Pinned host memory via cudaHostRegister
  • Separate CUDA stream for data transfers
  • cudaEvents for synchronization between compute and copy streams
  • Double buffering: while GPU computes batch N, CPU transfers batch N+1
  • Single cudaMemcpy sync per epoch for metrics

Pipeline Diagram:

Time →
Stream 0 (Compute): [Forward B0][Backward B0]      [Forward B1][Backward B1]
Stream 1 (Copy):                  [Copy B1 to GPU]                [Copy B2 to GPU]

Performance Tips

  1. Set cudaDeviceScheduleBlockingSync to prevent CPU core spinning
  2. Use batch sizes that are multiples of 256 for optimal warp utilization
  3. Prefer .to("cuda") once over repeated transfers
  4. The CachingAllocator makes temporary tensor creation nearly free

Build Configuration

Requirement Detail
Dependencies CUDA Toolkit, cuBLAS
CMake flags -O3 for both CXX and CUDA
Architecture Set CMAKE_CUDA_ARCHITECTURES to your GPU's compute capability
Supported sm_70 (Volta), sm_75 (Turing), sm_80 (Ampere), sm_89 (Ada), sm_100 (Blackwell)
Warning: Always compile CUDA code with -O3. Without it, the nvcc compiler disables register optimization and loop unrolling, causing up to 10x slowdowns as variables spill to global memory.
